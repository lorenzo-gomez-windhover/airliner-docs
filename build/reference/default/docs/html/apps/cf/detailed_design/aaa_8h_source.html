14  * My philosophy with this module is different than usual.
15  * My usual approach is to make lower-level modules as close to general-
16  * purpose utilities as practical. Instead, this module is tightly coupled
17  * to the needs of the state table modules (so that the state table logic
18  * is easier to grasp).
19  * The purpose of this module is to provide the details for carrying out
20  * many conceptually straightforward actions that are called out within
21  * the CFDP state tables (see s1.c, s2.c, r1.c, and r2.c).
22  * All of the routines in this module work on one state machine at a time.
23  * In most cases, a pointer to the state machine is supplied by the caller.
24  * In many cases, the routine interface could have been more specific
25  * (instead of passing the entire state machine), but that would have made
26  * the state table logic less readable.
27  * CHANGES:
