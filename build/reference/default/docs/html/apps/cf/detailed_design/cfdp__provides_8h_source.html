254 /* Each routine in this group expresses a given data structure as a
255  * character string. (The exact string format is not specified)
256  * WARNING: All returns from a particular routine point to the same
257  * character string. In other words, each call to a particular routine
258  * writes over the result of the previous call to that same routine.
259  * Therefore, be sure to copy (or print) the result from one call before
260  * making another call to the same routine.
261  * NOTES:
262  * 1) Each returned character string may be up to 'MAX_AS_STRING_LENGTH'
263  * bytes long.
264  * 2) When an ID is expressed as a string, it is in 'dotted-decimal' format.
265  * Each byte's value is expressed as a decimal number, and a dot is
266  * put between decimal numbers. Examples: "10" "101.151"
267  * 3) When a TRANSACTION is expressed as a string, the format is
268  * id-underscore-transaction_sequence_number. Examples:
269  * "10_1" -- transaction number 1 at CFDP entity 10.
270  * "101.151_803" -- transaction number 803 at CFDP entity 101.151.
271  */

308  * is '1', 'bytes_transferred' indicates how many bytes of file-data

311  * Otherwise, the return-status is '0', and the data returned
312  * is meaningless.
313  * NOTES: If the local CFDP node is the sender, then 'bytes-transferred'

315  * If the local CFDP node is the receiver, then 'bytes-transferred'
316  * indicates the number of bytes received.
317  * If the local CFDP node is the receiver, the 'file_size' will be
318  * unknown until a Metadata or EOF pdu is received. If the file-size
319  * is unknown, it will be given the value of zero.
320  */

324 /* WHAT IT DOES: If the specified transaction exists, then the return-status

326  * given transaction. Otherwise, the return-status is '0', and the
327  * data returned is meaningless.
328  * INPUTS: First, which transaction is of interest? Second, a pointer to

391 /* Caution: These routines are for a special purpose. They allow the
392  * engine's implementation of the CFDP protocol to be disabled during the
393  * initial Filedata transmissions and then resumed for the remainder of the
394  * protocol.
395  * Context: Suppose you had to transfer files at a much higher datarate
396  * than could be accomplished with a software implementation of CFDP
397  * running on a flight processor (say, 100 megabits/second). You could
398  * disable the engine's normal protocol while hardware blasts the initial
399  * transmission of the file, then re-enable the engine to send the EOF
400  * and respond to acks and naks.
401  * Note: If the engine does not output Filedata PDUs, it will not calculate
402  * a file checksum. If external file transfer is used, the engine user
403  * must calculate the file checksum and give it to the engine (by calling
404  * the routine cfdp_set_file_checksum).
405  */
406 
407 boolean cfdp_open_external_file_xfer (TRANSACTION trans);
408 /* WHAT IT DOES: For one particular transaction, tells the engine to disable
409  * its output of Filedata PDUs.
410  * INPUT: Which transaction?
411  * RETURN STATUS: '1' indicates acceptance. '0' indicates rejection.
412  * Acceptance will only occur if the given transaction is known to the
413  * engine, the engine's role is 'Sender', and the engine has not sent
414  * an EOF PDU.
415  * NOTES:
416  * 1) This routine should be called in response to the 'Transaction'
417  * Indication (or, in response to the 'Metadata_Sent' Indication).
418  * 2) The engine still responds to Cancel Requests and Finished-Cancel PDUs
419  * for the given transaction.
420  */
421 
422 boolean cfdp_close_external_file_xfer (TRANSACTION trans);
423 /* WHAT IT DOES: For one particular transaction, tells the engine that the
424  * engine user is done sending Filedata PDUs. The engine resumes normal
425  * implementation of the CFDP protocol (it sends the EOF PDU and responds
426  * to acks and naks).
427  * INPUT: Which transaction?
428  * RETURN STATUS: 1' indicates acceptance. '0' indicates rejection.
429  * Acceptance will only occur if the given transaction is known to the
430  * engine, and external file transfer is open for that transaction.
431  */
432
